# 2. 좋은 코드의 기준 - "응집도 중심"으로 바라보기

>Next.js(App Router)의 철학 : Server-first + 도메인 단위 응집도
>좋은 코드란 , 변화가 하나의 맥락 안에서 일어나는 코드다.

1. Server-fist 원칙
   - 가능한 모든 UI렌더링과 데이터 획득(패칭)을 서버에서 수행
   - 
1. JS번들 최소화 : 불필요한 클라이언트 전송 금지
2. 데이터로직 (Fetch)와 UI(Component)의 변화 시점을 묶는다.(응집도 중심) 

>"Server Component는 fetch와 render의 결합을 통해 도메인 단위의 응집을 이룬다."


# 3. 도메인 설정 - 책 상품 목록 예시
도메인 : 책 상품 목록 페이지 렌더링
요구사항 
- 사용자는 책 리스트를 보고 정렬,검색 할 수 있다.
- 클릭 시 상세페이지로 이동
제약조건
- 데이터 제약 : 책 개수 적고 변동 적음
- 비즈니스 제약 : SEO중요 , 초기 렌더 속도 중요
- 기술 제약 : Next.js App Router , Supbase API 사용
-> 클라이언트 상태관리 최소화 , 번들 최소화
-> 이 요구는 서버에서 데이터와 UI를 한 번에 처리하는 구조에 적합

# 6. 프롬프트 규칙화 - Rules VS Prompt


| 구분      | Rules로 고정                                     | Prompt로 남김       |
| ------- | --------------------------------------------- | ---------------- |
| 기술철학    | Server-first , 도메인응집(fetch + UI)<br>Clean FSD | -                |
| 품질 기준   | 번들 최소 , SEO 대응 , 상태처리                         | -                |
| 비즈니스 로직 |                                               | 정렬,검색 등 요구별 기능   |
| 제약조건    | React Query등 금지                               | -                |
| 구현 세부   | -                                             | UI/컴포넌트 구조 , 네이밍 |
- Rules는 항상 지켜야 하는 원칙이고 , Prompt는 문제별 설계 맥락
- AI코딩은 결국 Rule 기반 시스템 + Context특화 Prompt의 조합
