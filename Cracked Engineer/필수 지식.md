# 1. 기획 , 비즈니스 

- **MVP (Minimum Viable Product)**: 최소 기능 제품. 가설을 검증하기 위한 최소한의 핵심 기능만 담은 
  버전. '완벽'보다 '속도'와 '피드백'이 중요할 때 사용.
    
- **PMF (Product-Market Fit)**: 제품-시장 적합성. 우리가 만든 솔루션이 시장의 문제를 실제로 해결하고 있는지 판단하는 척도
    
- **Unit Economics**: 고객 한 명을 유치하는 비용(CAC) 대비 고객 한 명이 평생 가져다주는 이익(LTV)의 구조를 이해하는 것. 엔지니어가 클라우드 비용을 최적화해야 하는 이유와 직결됨.
    
- **User Story & Acceptance Criteria (AC)**: 사용자 입장에서 기능을 정의하고, 그 기능이 완료되었다고 판단할 구체적인 수용 기준. 기획자와 소통할 때 가장 중요한 기준점.
      
- **Lean Startup** : buiild , measure , Learn 과정 빠르게 반복. MVP 기능을 배포한 뒤 사용자의 반응을 보고 다음 기능을 결정
    
- **User Retention** : 사용자가 서비스를 이탈하지 않고 계속 사용하는 비율. push 기능을 추가해 사용자가 앱을 다시 사용하게끔 유도
      
- **Unit Economics** : 한 명의 사용자를 획득하는 비용 대비 그 사용자가 주는 가치



# 2. 설계 및 아키텍쳐
- **Scalability (확장성)**:
    
    - **Vertical Scale (Scale-up)**: 서버 사양을 높임.
        
    - **Horizontal Scale (Scale-out)**: 서버 대수를 늘림.
        
- **Latency vs Throughput**:
    
    - **Latency (지연 시간)**: 요청 하나가 처리되는 속도 (사용자 경험).
        
    - **Throughput (처리량)**: 단위 시간당 처리할 수 있는 요청 수 (시스템 용량).
        
- **DDD (Domain-Driven Design)**: 도메인 주도 설계. 비즈니스 복잡도를 코드로 그대로 옮기기 위해 비즈니스 로직(도메인)을 중심으로 설계하는 방법론.
    
- **CAP Theorem**: 분산 시스템에서 일관성(Consistency), 가용성(Availability), 분할 내성(Partition Tolerance) 세 가지를 동시에 만족할 수 없다는 원칙. 비즈니스 성격에 따라 무엇을 포기할지 결정해야 함.


# 3. 개발 및 엔지니어링

- **CI/CD (Continuous Integration/Deployment)**: 지속적 통합 및 배포. 코드가 작성되자마자 자동으로 테스트되고 서버에 반영되는 파이프라인
    
- **TDD (Test-Driven Development)**: 테스트 주도 개발. 실패하는 테스트 코드를 먼저 짜고 이를 통과하는 코드를 작성하여 코드의 안정성을 확보.
    
- **Technical Debt (기술 부채)**: 빠른 배포를 위해 타협한 품질 낮은 코드. 적절한 부채는 전략적이지만, 상환하지 않으면 개발 속도가 마비됨.
    
- **Observability (관측 가능성)**: 로그(Log), 메트릭(Metric), 트레이싱(Tracing)을 통해 시스템 내부 상황을 파악하고 장애를 선제적으로 대응하는 능려


# 4. 기술적 용어


### . 프레임워크 및 도구 관련 용어

- **Next.js**: 웹사이트를 만들기 위한 '도구 모음(프레임워크)'입니다. 단순히 화면을 그리는 것을 넘어, 서버와의 통신이나 페이지 이동 등을 효율적으로 처리할 수 있게 도와줌.
    
- **App Router (M-라우터)**: Next.js에서 폴더 구조를 이용해 웹페이지의 경로(URL)를 관리하는 최신 방식. 영상에서는 이를 통해 구조적인 설계를 강조.
    
- **리액트 쿼리 (React Query)**: 웹사이트에서 데이터를 가져오고, 캐싱(임시 저장)하고, 업데이트하는 작업을 쉽게 도와주는 라이브러리. 주로 클라이언트(사용자 브라우저) 쪽에서 데이터를 관리할 때 사용.
    

### 2. 렌더링 및 아키텍처 관련 용어

- **Server-first (서버 퍼스트)**: "가능한 한 모든 일을 서버에서 먼저 처리하자"는 철학입니다. 사용자의 컴퓨터(브라우저)가 힘겹게 계산하게 하지 말고, 성능 좋은 서버가 미리 다 그려서 보내주자는 전략.
    
- **UI 렌더링 (Rendering)**: 코드로 된 데이터를 사람이 볼 수 있는 예쁜 화면(HTML)으로 변환하는 과정.
    
- **SSR (Server-Side Rendering)**: 서버에서 화면을 미리 다 만들어서 사용자에게 보내주는 방식. 초기 로딩 속도가 빠르고 검색 엔진에 노출되기 유리
    
- **CSR (Client-Side Rendering)**: 서버는 최소한의 뼈대만 보내고, 사용자의 브라우저에서 자바스크립트를 실행해 화면을 그리는 방식
    
- **SEO (Search Engine Optimization, 검색 엔진 최적화)**: 네이버나 구글 같은 검색 엔진이 내 웹사이트 내용을 잘 읽어갈 수 있도록 만들어, 검색 결과 상단에 나오게 하는 기술
    

### 3. 컴포넌트 및 성능 관련 용어

- **서버 컴포넌트 (Server Components)**: 서버에서만 실행되는 화면 단위. 브라우저로 전달되는 자바스크립트 양이 줄어들어 웹사이트가 가벼워짐.
    
- **클라이언트 컴포넌트 (Client Components)**: 버튼 클릭, 마우스 드래그 등 사용자와 실시간으로 상호작용하기 위해 브라우저에서 실행되어야 하는 화면 단위.
    
- **JS 번들 (JavaScript Bundle)**: 웹사이트가 작동하기 위해 브라우저로 전송되는 자바스크립트 파일들의 묶음. 이 뭉치가 커질수록 웹사이트 로딩이 느려지므로, 최소화해야 한다.
    
- **데이터 패칭 (Data Fetching)**: 서버나 데이터베이스에서 필요한 정보(예: 책 목록, 가격 등)를 가져오는 행위

### 4. 설계 원칙 관련 용어

- **도메인 단위 응집도 (Domain Cohesion)**: 비슷한 목적을 가진 기능들끼리 모아두는 것. 예를 들어 '책'에 대한 데이터 로직과 '책'을 보여주는 화면 코드를 한곳에 두면, 나중에 수정할 때 찾기 쉽고 관리가 용이해짐.
    
- **인터랙티비티 (Interactivity)**: 사용자가 버튼을 누르거나 입력창에 글을 쓰는 등 웹사이트와 '상호작용'하는 성질을 뜻합니다.
    
- **엔티티 (Entities)**: 데이터의 최소 단위나 객체를 의미합니다. 영상에서는 `entities` 폴더 하위에 관련 로직을 모으는 구조를 제안.

- **Atomic Deployment** : 배포 중 오류 발생 시 이전 버전이 유지되는 방식
  Vercel은 기본적으로 지원하며, 무중단 배포를 보장.
- **Build Caching**변경되지 않은 파일은 다시 빌드하지 않는 기술
  Turbopack과 Vercel 캐시를 활용해 빌드 시간을 단축.
- **Rollback Strategy**배포된 코드가 문제가 생겼을 때 즉시 되돌리는 전략
  클릭 한 번으로 이전 빌드 버전으로 복구하는 능력을 익혀야 함.
